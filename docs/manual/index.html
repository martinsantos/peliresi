1. Vision General

1.1 Que es SITREP

SITREP (Sistema de Trazabilidad de Residuos Peligrosos) es la plataforma digital del Gobierno de Mendoza para el seguimiento completo de manifiestos de residuos peligrosos. Desarrollado para la Direccion General de Fiscalizacion Ambiental (DGFA), permite digitalizar y automatizar el proceso completo de gestion de residuos peligrosos en la provincia.

El sistema permite el monitoreo en tiempo real desde la generacion del residuo, pasando por su transporte, hasta el tratamiento y disposicion final, reemplazando los procesos basados en papel con un sistema electronico conforme a la Ley Nacional 24.051 de Residuos Peligrosos y su Decreto Reglamentario 831/93.

Procesos Cubiertos

SITREP abarca la totalidad del ciclo de vida de los residuos peligrosos:

  • Creacion de manifiestos electronicos y firma digital: Los generadores crean manifiestos en formato electronico con todos los datos normativos requeridos (tipo de residuo, cantidad, transportista, operador destino) y los firman electronicamente para dar inicio al proceso.
  • Seguimiento GPS en tiempo real del transporte: Durante el transporte, el sistema captura la posicion GPS del vehiculo cada 30 segundos, permitiendo a todos los actores visualizar la ubicacion del residuo en un mapa interactivo.
  • Verificacion de peso y documentacion de discrepancias: Al llegar a la planta del operador, se registra el peso real del residuo y se compara automaticamente con el peso declarado, documentando cualquier diferencia significativa.
  • Registro de tratamiento y certificados de disposicion: El operador registra el metodo de tratamiento aplicado conforme al catalogo autorizado y, al completar el proceso, el sistema genera el Certificado de Tratamiento y Disposicion Final en formato PDF.
  • Reportes estadisticos y cumplimiento regulatorio: El sistema genera reportes detallados sobre manifiestos, residuos tratados, transporte, generadores, operadores, tratamientos, departamentos y mapas de actores, con exportacion a PDF y CSV.

Categorias de Residuos (Corrientes Y)

La Ley 24.051 clasifica los residuos peligrosos en categorias sometidas a control (Anexo I), que van desde Y1 (desechos clinicos) hasta Y48. El catalogo de SITREP mantiene todas las categorias activas configurables por el administrador, con su codigo, nombre, descripcion, caracteristicas de peligrosidad y estado (activo/inactivo).

Obligaciones de los Actores segun Normativa

Generador

Obligado a inscribirse en el Registro Nacional de Generadores y Operadores de Residuos Peligrosos. Debe adoptar medidas para disminuir la cantidad de residuos peligrosos, separar adecuadamente los residuos, envasar, etiquetar y mantener un registro de operaciones. Es responsable del residuo hasta su tratamiento o disposicion final.

Transportista

Debe poseer habilitacion especifica para el transporte de residuos peligrosos. Los vehiculos deben cumplir requisitos tecnicos de seguridad, contar con seguro obligatorio y el chofer debe estar capacitado en manejo de sustancias peligrosas. El manifiesto de transporte debe acompanar al residuo durante todo el trayecto.

Operador

Planta de tratamiento y/o disposicion final habilitada por la autoridad competente. Debe contar con inscripcion vigente, equipamiento adecuado para los tipos de residuos autorizados y personal capacitado. Emite el certificado de disposicion final que cierra el ciclo del manifiesto.

2. Guia Rapida por Rol

2.1 Administrador DGFA -- Primeros Pasos

Rol: Administrador DGFA — Acceso completo a todos los modulos del sistema.

  1. Ingresar a sitrep.ultimamilla.com.ar con credenciales de administrador.
  2. El Dashboard muestra KPIs principales: manifiestos activos, en transito, completados, alertas pendientes.
  3. Navegar a Centro de Control para ver el mapa de transportes activos en tiempo real.
  4. Consultar Reportes con 8 pestanas de analisis (manifiestos, residuos, transporte, generadores, operadores, tratamientos, departamentos, mapa de actores).
  5. Gestionar actores desde Admin > Actores (generadores, transportistas, operadores).
  6. Configurar reglas de alerta desde Alertas.
  7. Revisar el log de auditoria inmutable desde Admin > Auditoria.

Consejos para el Administrador

  • Centro de Control como monitor principal: Mantener el Centro de Control abierto en una pantalla dedicada para supervision continua. El auto-refresco de 30 segundos mantiene los datos actualizados sin intervencion.
  • Alertas proactivas: Configurar reglas de alerta para diferencias de peso mayores al 10%, viajes que excedan el tiempo estimado, y manifiestos estancados en un estado por mas de 48 horas.
  • Reportes periodicos: Utilizar el preset "Ultimos 30 dias" para generar informes mensuales. Exportar a CSV para analisis avanzado en planillas de calculo.
  • Auditoria regular: Revisar periodicamente el log de auditoria filtrando por acciones criticas (eliminacion, cambio de rol, cancelacion de manifiestos).
  • Carga masiva inicial: Al incorporar nuevos actores al sistema, utilizar la herramienta de carga masiva con la plantilla CSV descargable para evitar errores de carga manual.

2.2 Generador -- Crear mi Primer Manifiesto

Rol: Generador — Empresa productora de residuos peligrosos.

  1. Ingresar con credenciales de generador.
  2. Desde el Dashboard, hacer clic en "Nuevo Manifiesto".
  3. Completar los datos: tipo de residuo (categorias Y conforme Ley 24.051), cantidad y unidad de medida.
  4. Seleccionar el transportista habilitado de la lista desplegable.
  5. Seleccionar el operador destino (planta de tratamiento autorizada).
  6. Revisar el resumen completo y firmar electronicamente.
  7. El sistema genera automaticamente el codigo QR y notifica al transportista asignado.
  8. Seguir el estado del manifiesto desde Manifiestos: ver timeline de eventos y ubicacion GPS en tiempo real.

Consejos para el Generador

  • Verificar datos antes de firmar: Una vez firmado, el manifiesto pasa a APROBADO y no puede editarse. Revisar cuidadosamente tipo de residuo, cantidad, transportista y operador en el paso 5 de revision.
  • Usar el codigo QR: Al aprobar el manifiesto, se genera un QR unico. Imprimir y adjuntar al envase del residuo para facilitar la trazabilidad fisica.
  • Monitorear en tiempo real: Una vez que el transportista confirma el retiro, es posible seguir la ubicacion del vehiculo en el mapa GPS desde la pagina de detalle del manifiesto.
  • Descargar el certificado: Cuando el operador cierra el manifiesto (estado TRATADO), descargar el Certificado de Disposicion Final como respaldo documental del cumplimiento normativo.
  • Revisar notificaciones: El sistema envia notificaciones automaticas en cada cambio de estado. Mantener el navegador abierto o instalar la PWA para recibir actualizaciones.

2.3 Transportista -- Mi Primer Viaje

Rol: Transportista — Transportista habilitado para residuos peligrosos.

  1. Ingresar con credenciales de transportista (web o app movil).
  2. Ver los manifiestos asignados en el Dashboard (en mobile aparece un banner de viaje destacado).
  3. Al llegar al generador: Confirmar Retiro → se activa el GPS automaticamente.
  4. Durante el transporte: el GPS envia la posicion cada 30 segundos al servidor.
  5. Es posible pausar y reanudar el tracking si se realizan paradas intermedias.
  6. Si hay incidentes durante el viaje: Registrar Incidente con descripcion detallada.
  7. Al llegar al operador: Confirmar Entrega → el GPS se desactiva automaticamente.
  8. Escanear el codigo QR con la camara para acceso rapido al manifiesto.

Consejos para el Transportista

  • Instalar la PWA: Para la mejor experiencia en campo, instalar SITREP como aplicacion en el telefono desde https://sitrep.ultimamilla.com.ar/app/. Esto habilita el modo offline y acceso rapido desde la pantalla de inicio.
  • Permisos GPS: Al confirmar el retiro, el navegador solicitara permisos de ubicacion. Es imprescindible aceptar para que el tracking funcione correctamente. Si se deniegan por error, ir a configuracion del navegador para habilitarlos manualmente.
  • Mantener la aplicacion abierta: El GPS se envia cada 30 segundos mientras la aplicacion esta activa. Si se cierra o se pone en segundo plano, los puntos pendientes se guardan localmente y se envian al reabrir la app.
  • Usar pausa en paradas: Si se realizan paradas intermedias (combustible, descanso), usar el boton de Pausa para registrar el incidente. Reanudar al continuar el viaje.
  • Registrar incidentes inmediatamente: Ante cualquier incidencia durante el transporte (accidente, demora, desvio de ruta), registrar el incidente con descripcion detallada lo antes posible.
  • Escanear QR al llegar: Al llegar a la planta del operador, escanear el QR del manifiesto para verificar rapidamente los datos antes de confirmar la entrega.

2.4 Operador -- Recibir y Tratar Residuos

Rol: Operador — Planta de tratamiento y disposicion final.

  1. Ingresar con credenciales de operador.
  2. Ver los manifiestos entrantes en el Dashboard.
  3. Al recibir la carga: Confirmar Recepcion.
  4. Registrar el pesaje real y comparar con el peso declarado por el generador.
  5. Si hay diferencias significativas de peso: se documentan automaticamente en el sistema.
  6. Si todo esta conforme: Firmar Recepcion Conforme → el manifiesto pasa a estado RECIBIDO.
  7. Si la carga no es aceptable: Rechazar Carga indicando el motivo del rechazo.
  8. Registrar el tratamiento aplicado (seleccionar metodo del catalogo autorizado).
  9. Cerrar Manifiesto → el sistema genera el Certificado de Disposicion Final (PDF descargable).

Consejos para el Operador

  • Verificar manifiestos entrantes: Revisar en el Dashboard los manifiestos con estado ENTREGADO que estan esperando recepcion en la planta. Cada minuto de demora en la confirmacion retrasa la cadena de trazabilidad.
  • Pesaje inmediato: Registrar el peso real apenas se reciba el residuo. El sistema calcula automaticamente las diferencias con el peso declarado y las documenta para la auditoria.
  • Motivos de rechazo claros: Si se rechaza una carga, incluir un motivo detallado y especifico (ej: "Residuo no coincide con categoria Y declarada", "Documentacion de transporte incompleta", "Excede capacidad autorizada").
  • Seleccionar tratamiento correcto: Al registrar el tratamiento, asegurarse de seleccionar el metodo del catalogo que corresponde al tipo de residuo recibido. Solo aparecen los metodos autorizados para la planta.
  • Conservar certificados: Al cerrar el manifiesto, descargar y archivar el Certificado de Disposicion Final. Este documento es la prueba legal de que el residuo fue tratado conforme a la normativa vigente.

3.2 Manifiestos

Modulo central del sistema para la gestion completa del ciclo de vida de los manifiestos de residuos peligrosos.

Listado (/manifiestos)

  • Busqueda de texto completo por numero de manifiesto.
  • Filtro por estado (dropdown con los 10 estados posibles).
  • Tabla paginada con controles de pagina y limite por pagina.
  • Clic en cualquier fila para acceder al detalle del manifiesto.

Crear Nuevo (/manifiestos/nuevo)

Formulario multi-paso guiado:

  1. Paso 1: Seleccion de tipo de residuo del catalogo de categorias Y.
  2. Paso 2: Cantidad y unidad de medida.
  3. Paso 3: Seleccion de transportista (auto-completa CUIT, telefono, domicilio, habilitacion).
  4. Paso 4: Seleccion de operador destino (auto-completa datos).
  5. Paso 5: Revision del resumen y firma electronica.

El manifiesto se guarda como BORRADOR hasta que se firma electronicamente.

Detalle (/manifiestos/:id)

  • Tarjeta completa con todos los datos del manifiesto.
  • Timeline cronologico de todos los eventos (EventoManifiesto).
  • Botones de accion que cambian dinamicamente segun el estado actual Y el rol del usuario (mas de 10 acciones posibles).
  • Mapa GPS para manifiestos en estado EN_TRANSITO.
  • Boton de descarga PDF (disponible en todos los estados).
  • Boton de descarga de Certificado de Disposicion (solo en estado TRATADO).

Workflow de Estados

BORRADOR → APROBADO → EN_TRANSITO → ENTREGADO → RECIBIDO → EN_TRATAMIENTO → TRATADO
ENTREGADO → RECHAZADO
CUALQUIER ESTADO (excepto CANCELADO/TRATADO) → CANCELADO

Nota: INCIDENTE es un evento registrado durante EN_TRANSITO, no un estado terminal. El manifiesto permanece en estado EN_TRANSITO mientras se resuelve el incidente.

Datos del Manifiesto

Cada manifiesto contiene la siguiente informacion:

  • Numero de manifiesto: Identificador unico generado por el sistema.
  • Generador: Razon social, CUIT, domicilio, numero de inscripcion.
  • Tipo de residuo: Categoria Y conforme Ley 24.051, con caracteristicas de peligrosidad.
  • Cantidad y unidad: Peso o volumen declarado del residuo.
  • Transportista: Empresa, CUIT, habilitacion, vehiculo asignado, chofer.
  • Operador destino: Planta, CUIT, habilitacion, metodo de tratamiento autorizado.
  • Fechas: Creacion, aprobacion, retiro, entrega, recepcion, tratamiento, cierre.
  • Codigo QR: Generado automaticamente al aprobar para acceso rapido via escaneo.
  • Timeline de eventos: Historial completo de cada accion realizada sobre el manifiesto.
  • Certificacion Blockchain: Registro inmutable del hash SHA-256 en la red Ethereum (ver seccion 3.10).

El Centro de Control calcula y muestra:

3.10 Certificacion Blockchain

SITREP incluye un modulo de certificacion blockchain que permite registrar manifiestos en la red Ethereum como prueba inmutable de integridad. La certificacion genera un hash criptografico (SHA-256) del contenido del manifiesto y lo almacena en un smart contract publico, creando un sello temporal verificable por cualquier persona.

Nota: La certificacion blockchain no reemplaza la firma digital regulada por la normativa argentina (ver CU-S10). Es un mecanismo complementario de integridad de datos que garantiza que el contenido del manifiesto no fue alterado despues de su certificacion.

3.10.1 Red y Contrato Inteligente

El sistema opera sobre Ethereum Sepolia, una red de prueba (testnet) publica y gratuita que es identica en funcionamiento a la red principal de Ethereum (mainnet). La testnet permite validar completamente la tecnologia sin incurrir en costos de gas reales.

La certificacion es bajo demanda: el usuario hace clic en “Certificar en Blockchain” desde la pagina de detalle del manifiesto. No es automatica.

3.10.3 Datos Certificados (Hash Canonico)

El hash SHA-256 se calcula sobre un JSON canonico con los siguientes campos, siempre en el mismo orden:

CampoDescripcion
numeroNumero del manifiesto (ej: 2026-000147)
generadorIdID del generador
generadorCuitCUIT del generador
transportistaIdID del transportista
transportistaCuitCUIT del transportista
operadorIdID del operador
operadorCuitCUIT del operador

Los residuos se ordenan alfabeticamente por tipoResiduoId. El resultado es determinista: los mismos datos siempre generan el mismo hash SHA-256 de 64 caracteres hexadecimales.

Estrategia de Cache

  1. Primero intenta obtener el recurso de la red.
  2. Si la red no esta disponible, busca en el cache local.
  3. Si el recurso no esta en cache, muestra la pagina de fallback offline.
  4. Las solicitudes a la API (/api/) NO se cachean para garantizar datos actualizados.
  5. Los recursos estaticos se cachean automaticamente al ser visitados.

Autenticacion y Autorizacion

El flujo de autenticacion sigue el patron estandar JWT con refresh tokens:

  1. Login: El usuario envia email y contrasena a POST /api/auth/login.
  2. Validacion: El servidor verifica las credenciales con bcrypt y, si son validas, genera un access token (duracion 24 horas) y un refresh token (duracion 7 dias).
  3. Solicitudes autenticadas: Cada solicitud incluye el access token en el header Authorization: Bearer {token}.
  4. Middleware de autorizacion: Cada endpoint verifica el rol del usuario contra los roles permitidos para esa operacion.
  5. Renovacion automatica: Cuando el access token expira, el frontend envia automaticamente el refresh token a POST /api/auth/refresh para obtener un nuevo par de tokens.
  6. Revocacion: Los refresh tokens se almacenan en base de datos y pueden ser revocados individualmente.

6.1 Diagrama de Estados

El workflow de manifiestos es el corazon del sistema SITREP. Cada manifiesto sigue un ciclo de vida definido con transiciones controladas por estado actual y rol del usuario.

El flujo completo desde la creacion hasta el tratamiento del residuo peligroso:

BORRADOR → APROBADO → EN_TRANSITO → ENTREGADO → RECIBIDO → EN_TRATAMIENTO → TRATADO

El operador puede rechazar la carga cuando:

  • El residuo no coincide con la categoria Y declarada en el manifiesto.
  • La cantidad entregada excede significativamente lo declarado.
  • La documentacion de transporte esta incompleta o incorrecta.
  • El residuo excede la capacidad autorizada de la planta.
  • El envase o embalaje no cumple las condiciones de seguridad requeridas.
CUALQUIER ESTADO → CANCELADO (excepto CANCELADO y TRATADO)

La cancelacion puede ser realizada por el generador o el administrador en cualquier momento del ciclo, excepto cuando el manifiesto ya esta CANCELADO o TRATADO. Requiere un motivo documentado.

Durante el transporte, el transportista puede registrar incidentes sin cambiar el estado del manifiesto. Los tipos de incidente incluyen:

  • PAUSA: Parada intermedia planificada (descanso, carga de combustible).
  • REANUDACION: Reanudacion del viaje tras una pausa.
  • ACCIDENTE: Accidente vial o incidente de seguridad.
  • DEMORA: Demora significativa por trafico, clima u otros factores.
  • DESVIO: Desvio de la ruta planificada.
  • OTRO: Cualquier otro incidente relevante con descripcion libre.

6.3 Flujo GPS

Flujo del seguimiento GPS durante el transporte de residuos peligrosos.

  1. Activacion: El transportista confirma el retiro en el generador. El estado cambia a EN_TRANSITO y se activa navigator.geolocation.watchPosition().
  2. Captura continua: El navegador reporta la posicion continuamente al obtener nuevas coordenadas.
  3. Transmision periodica: Cada 30 segundos, la ultima posicion conocida se envia al servidor via POST /api/manifiestos/:id/ubicacion.
  4. Datos transmitidos: latitud, longitud, velocidad (m/s), direccion (grados), precision (metros).
  5. Resiliencia offline: Si falla el envio, el punto GPS se agrega a una cola (maximo 5 puntos pendientes). La cola se persiste en localStorage con clave gps_pending_{manifiestoId}.
  6. Recuperacion: Al montar la aplicacion, los puntos GPS pendientes guardados en localStorage se envian automaticamente al backend.
  7. Pausa/Reanudacion: El transportista puede pausar el tracking, lo cual registra un incidente de tipo PAUSA. Al reanudar, se registra un incidente REANUDACION.
  8. Desactivacion: Al confirmar la entrega en el operador, el estado cambia a ENTREGADO y el watcher GPS se limpia correctamente (incluye listener beforeunload para prevenir watchers zombi).

Temporizador

El temporizador de viaje muestra el tiempo transcurrido desde la fechaRetiro (timestamp del servidor), no un contador local. Esto garantiza consistencia entre dispositivos y sesiones.

Cache Backend

El endpoint de ubicacion GPS utiliza un cache en memoria con TTL de 30 segundos por manifiestoId. Esto reduce la latencia de las escrituras GPS de ~250ms (consulta completa a BD) a ~24ms (cache hit). El cache se invalida cuando el manifiesto cambia de estado (por ejemplo, al confirmar entrega).

Maquina de Estados GPS

El componente de tracking GPS en el frontend implementa una maquina de estados para gestionar el ciclo de vida del GPS:

Formato de Datos GPS

Cada punto GPS enviado al servidor contiene:

{
  "latitud": -32.8895,       // Latitud en grados decimales
  "longitud": -68.8458,      // Longitud en grados decimales
  "velocidad": 45.2,         // Velocidad en m/s (opcional)
  "direccion": 180.5         // Rumbo en grados (0-360, opcional)
}

El servidor adiciona automaticamente: precision (metros), timestamp (ISO 8601), y el manifiestoId de la URL.

Resiliencia ante Perdida de Conexion

  1. El frontend intenta enviar la posicion GPS al servidor.
  2. Si falla (sin conexion, timeout, error 5xx), el punto se agrega a pendingUpdatesRef.
  3. La cola de pendientes tiene un limite maximo de 5 puntos (los mas antiguos se descartan para priorizar datos recientes).
  4. La cola se persiste en localStorage con clave gps_pending_{manifiestoId} para sobrevivir cierres de aplicacion.
  5. Al montar el componente de viaje, se verifica si hay puntos pendientes en localStorage.
  6. Si existen puntos pendientes, se envian al backend secuencialmente antes de iniciar el tracking normal.
  7. Adicionalmente, el hook useConnectivity realiza un health ping cada 30 segundos y, al detectar reconexion, ejecuta un flush de la cola de sincronizacion de IndexedDB.

8. Preguntas Frecuentes

Como accedo al sistema por primera vez?

El administrador del sistema (DGFA) debe crear su cuenta de usuario y asignarle el rol correspondiente (Generador, Transportista u Operador). Una vez creada la cuenta, acceda a https://sitrep.ultimamilla.com.ar/ con el email y contrasena proporcionados. Para la version mobile, acceda a https://sitrep.ultimamilla.com.ar/app/ desde el navegador de su telefono.

Que navegadores son compatibles?

SITREP es compatible con las versiones actualizadas de los principales navegadores:

  • Google Chrome: Version 90+ (recomendado para Android)
  • Safari: Version 15+ (recomendado para iOS)
  • Mozilla Firefox: Version 90+
  • Microsoft Edge: Version 90+

Para el tracking GPS y la instalacion PWA, se recomienda Google Chrome en Android y Safari en iOS.

Que hago si el GPS no funciona en mi telefono?

El tracking GPS requiere que se concedan permisos de ubicacion al navegador. Si el GPS no funciona:

  1. Verificar permisos: En la configuracion del navegador, asegurarse de que sitrep.ultimamilla.com.ar tenga permiso de ubicacion "Permitir".
  2. Verificar GPS del dispositivo: Asegurarse de que el GPS del telefono este activado (Configuracion > Ubicacion).
  3. Modo de alta precision: Activar el modo de ubicacion de alta precision que combina GPS, WiFi y datos moviles.
  4. Reiniciar el navegador: Cerrar completamente el navegador y volver a abrir la aplicacion.
  5. Verificar HTTPS: El GPS solo funciona en conexiones seguras (HTTPS). No funciona en HTTP ni en file://.

Si el problema persiste, el sistema almacena los puntos GPS en una cola local y los enviara automaticamente cuando se restablezca la conexion.

Puedo usar el sistema sin conexion a internet?

SITREP tiene soporte parcial para modo offline gracias a la tecnologia PWA:

  • Recursos estaticos: La aplicacion se cachea localmente y puede cargarse sin conexion.
  • Datos GPS: Los puntos GPS capturados sin conexion se almacenan en localStorage y se envian automaticamente al recuperar la conectividad.
  • Limitaciones: Las operaciones que requieren consulta al servidor (crear manifiestos, cambiar estados, ver reportes) no estan disponibles sin conexion. Se muestra una pagina de fallback offline.
  • Indicador: El componente ConnectivityIndicator muestra una barra de estado cuando se pierde la conexion.
Como instalo la aplicacion en mi telefono?

Android (Chrome):

  1. Abrir https://sitrep.ultimamilla.com.ar/app/ en Google Chrome.
  2. Esperar 45 segundos hasta que aparezca el banner de instalacion, o abrir el menu (tres puntos) y seleccionar "Agregar a pantalla de inicio".
  3. Confirmar la instalacion. La app aparecera como icono en el launcher.

iOS (Safari):

  1. Abrir https://sitrep.ultimamilla.com.ar/app/ en Safari (no Chrome).
  2. Tocar el boton Compartir (cuadrado con flecha hacia arriba).
  3. Seleccionar "Agregar a la pantalla de inicio".
  4. Confirmar el nombre y tocar Agregar.
Que es el codigo QR del manifiesto?

El codigo QR se genera automaticamente cuando un manifiesto es aprobado (firmado electronicamente). Contiene el identificador unico del manifiesto y permite:

  • Acceso rapido al detalle del manifiesto escaneando con la camara del telefono.
  • Verificacion de la identidad del manifiesto en campo (retiro, entrega, recepcion).
  • Trazabilidad fisica: se puede imprimir y adjuntar al envase del residuo.

Para escanear, acceder a la herramienta Escaner QR desde el menu de la aplicacion mobile, o usar cualquier lector de QR estandar.

Como exporto datos del sistema?

SITREP ofrece dos formatos de exportacion:

  • PDF: Generado en el navegador del cliente con jsPDF. Incluye header institucional, KPIs graficos, tablas formateadas y footer con paginacion. Disponible desde la seccion de Reportes.
  • CSV: Generado en el servidor. Descarga un archivo CSV con datos tabulares del periodo seleccionado. Limite de 10,000 filas por exportacion. Disponible para manifiestos, generadores, transportistas y operadores.

Ademas, cada manifiesto individual puede descargarse como PDF desde su pagina de detalle, y los manifiestos en estado TRATADO incluyen el Certificado de Disposicion Final como PDF separado.

Que sucede si hay diferencia de peso entre lo declarado y lo recibido?

Cuando el operador registra el peso real (cantidadRecibida) al recibir la carga, el sistema calcula automaticamente la diferencia con el peso declarado por el generador. Los escenarios posibles son:

  • Sin diferencia: El peso recibido coincide con el declarado (tolerancia configurable).
  • Diferencia menor (<5%): Se documenta automaticamente pero no genera alerta.
  • Diferencia significativa (>5%): Se documenta y puede generar una alerta automatica si hay una regla configurada para este evento.

En todos los casos, la diferencia queda registrada en el campo tipoDiferencia del manifiesto y en el log de auditoria, proporcionando trazabilidad completa para fiscalizacion.

Puedo cancelar un manifiesto despues de firmarlo?

Si, un manifiesto puede cancelarse en cualquier momento del ciclo, excepto cuando ya se encuentra en estado TRATADO o CANCELADO. La cancelacion la puede realizar el generador propietario del manifiesto o un administrador DGFA. Al cancelar, se requiere indicar un motivo y todos los actores involucrados son notificados.

Nota importante: Si el manifiesto esta EN_TRANSITO, la cancelacion desactiva el tracking GPS y el transportista es notificado inmediatamente.

Como funciona la auditoria del sistema?

Cada operacion realizada en SITREP genera un registro inmutable en la tabla de auditoria. El registro incluye:

  • La accion realizada (crear, editar, eliminar, cambiar estado).
  • El modulo afectado (manifiestos, actores, usuarios, etc.).
  • Los datos ANTES del cambio (JSON completo).
  • Los datos DESPUES del cambio (JSON completo).
  • La direccion IP del cliente.
  • El user agent (navegador/dispositivo).
  • El usuario responsable.
  • El timestamp exacto.

Los registros de auditoria no pueden ser modificados ni eliminados por ningun usuario del sistema, incluido el administrador. Esto garantiza la integridad del historial para fines de fiscalizacion y cumplimiento normativo.

Cuantos transportistas puede soportar el sistema simultaneamente?

La configuracion actual esta optimizada para 50 transportistas simultaneos, cada uno enviando posiciones GPS cada 30 segundos. Esto representa aproximadamente 100 solicitudes GPS por minuto en total, bien dentro del limite de rate limiting (600 req/min/IP) y de la capacidad del connection pool de PostgreSQL (40 conexiones activas).

El sistema esta disenado para escalar a 200 usuarios concurrentes mediante:

  • Aumento de instancias PM2 (de 2 a 4-6).
  • Migracion de cache en memoria a Redis.
  • Aumento del connection pool y max_connections de PostgreSQL.
  • Ajuste de rate limiting.

Ver la seccion 5.6 (Escalamiento a 200 Usuarios) para el plan detallado.

Que categorias de residuos maneja el sistema?

El sistema implementa el catalogo completo de categorias de residuos sometidos a control conforme al Anexo I de la Ley Nacional 24.051, con codigos desde Y1 hasta Y48. Cada categoria incluye:

  • Codigo Y: Identificador unico de la corriente (Y1, Y2, ..., Y48).
  • Nombre: Denominacion oficial de la corriente de residuos.
  • Descripcion: Detalle del tipo de residuo incluido en la categoria.
  • Caracteristicas de peligrosidad: Codigos H asociados (H1-H13).
  • Estado: Activo o inactivo (solo los activos aparecen al crear manifiestos).

El administrador puede activar/desactivar categorias segun las necesidades de la provincia, pero no puede crear codigos fuera del rango normativo establecido por la ley.

Donde se almacenan los datos del sistema?

Los datos del sistema se almacenan en una base de datos PostgreSQL alojada en un contenedor Docker en el servidor de produccion (IP: 23.105.176.45). La base de datos se denomina trazabilidad_rrpp.

Componentes de almacenamiento:

  • PostgreSQL (Docker): Datos transaccionales (manifiestos, actores, eventos, GPS, auditoria, notificaciones, alertas, usuarios).
  • Servidor de archivos: Frontend estatico servido por Nginx desde /var/www/sitrep/.
  • localStorage (cliente): Tokens de sesion, puntos GPS pendientes, estado de viaje, preferencias de usuario.
  • IndexedDB (cliente): Cola de sincronizacion para operaciones offline.
  • Service Worker cache (cliente): Recursos estaticos para funcionamiento offline (trazabilidad-rrpp-v4).
Como funciona el auto-refresco de datos?

SITREP utiliza un mecanismo de polling para mantener los datos actualizados sin necesidad de recargar la pagina:

  • Dashboard: Los KPIs y manifiestos recientes se actualizan cada 30 segundos via useDashboardStats con refetchInterval: 30000.
  • Centro de Control: Todos los datos (mapa, KPIs, pipeline, viajes) se recargan cada 30 segundos. Un countdown visible muestra el tiempo restante para el proximo refresco.
  • Notificaciones: El icono de campana consulta nuevas notificaciones cada 30 segundos.
  • Detalle de manifiesto EN_TRANSITO: La posicion GPS en el mapa se actualiza con cada nuevo punto recibido del transportista.

Nota tecnica: El polling se implementa con TanStack Query v5, que maneja automaticamente el cache, la deduplicacion de solicitudes simultaneas, y la pausacion del polling cuando la pestana esta inactiva.

Como se manejan las sesiones expiradas?

El sistema utiliza un mecanismo automatico de renovacion de sesion:

  1. El access token tiene una duracion de 24 horas.
  2. Cuando el access token expira, el frontend detecta un error 401 (Unauthorized).
  3. Automaticamente envia el refresh token (duracion 7 dias) a POST /api/auth/refresh.
  4. Si el refresh token es valido, el servidor genera un nuevo par de tokens.
  5. La solicitud original se reintenta con el nuevo access token.
  6. Si el refresh token tambien expiro (mas de 7 dias sin usar el sistema), se redirige al login.

Sincronizacion cross-tab: Si el usuario tiene multiples pestanas abiertas, la renovacion de tokens se sincroniza via el evento storage de localStorage para evitar solicitudes de refresh concurrentes que podrian agotar la cola de renovacion.

Como se protege el sistema contra ataques de fuerza bruta?

El sistema implementa multiples capas de proteccion:

  • Rate Limiting: Limite de 600 solicitudes por minuto por direccion IP. Al superar el limite, las solicitudes se rechazan con error 429 (Too Many Requests).
  • bcrypt: Las contrasenas se almacenan como hash bcrypt, lo que hace computacionalmente inviable su recuperacion por fuerza bruta.
  • JWT: Los tokens son firmados criptograficamente, lo que impide su falsificacion.
  • Refresh Token en BD: Los refresh tokens se almacenan en base de datos y pueden ser revocados individualmente.
  • Helmet: Headers de seguridad que previenen ataques comunes (XSS, clickjacking, sniffing).
  • Trust Proxy: Configurado para detectar la IP real del cliente detras de Nginx, lo que permite que el rate limiting funcione correctamente incluso con proxies.
El sistema cumple con la normativa de proteccion de datos?

SITREP implementa multiples capas de proteccion de datos:

  • Autenticacion segura: JWT con access tokens de vida corta (24h) y refresh tokens (7d) almacenados en base de datos.
  • Contrasenas: Hash con bcrypt (algoritmo irreversible).
  • Comunicacion cifrada: HTTPS con certificado SSL Let's Encrypt en todas las comunicaciones.
  • Rate limiting: Proteccion contra ataques de fuerza bruta (600 req/min/IP).
  • Headers de seguridad: Helmet configura CSP, HSTS, X-Frame-Options y otros headers protectivos.
  • CORS restrictivo: Solo acepta solicitudes del dominio de produccion.
  • Auditoria inmutable: Registro completo de toda operacion con datos before/after.
  • Control de acceso por roles: Cada endpoint verifica el rol del usuario antes de ejecutar la operacion.

10.3 Procedimiento: Importacion Masiva de Datos

Procedimiento detallado para importar actores mediante archivos CSV:

Preparacion del Archivo

  1. Acceder a Admin > Carga Masiva.
  2. Seleccionar el tipo de actor a importar (Generadores, Transportistas u Operadores).
  3. Hacer clic en "Descargar Plantilla" para obtener el CSV con las columnas requeridas.
  4. Abrir el archivo en un editor de hojas de calculo (Excel, Google Sheets, LibreOffice Calc).
  5. Completar los datos respetando el formato de cada columna.

Consideraciones del Formato CSV

  • Codificacion: UTF-8 (para soporte de caracteres especiales como tildes y enie).
  • Separador: Coma (,) como separador de campos.
  • Campos con comas: Deben estar entrecomillados con comillas dobles ("campo con, coma").
  • Campos multilinea: Soportados dentro de comillas dobles (RFC 4180).
  • Formato CUIT: XX-XXXXXXXX-X (con guiones).
  • Coordenadas: Latitud y longitud en formato decimal (ej: -32.8895, -68.8458).
  • Codigos Y: Los separadores pueden ser inconsistentes. El sistema acepta coma, barra, "e" y espacios.
  • Certificados: Algunos usan guion largo (en-dash, U+2013) en lugar de guion comun. El sistema los normaliza automaticamente.

Proceso de Carga

  1. Hacer clic en "Subir Archivo" y seleccionar el CSV completado.
  2. El sistema parsea el archivo y muestra una vista previa.
  3. Las filas validas se marcan en verde; las filas con errores en rojo con detalle del error.
  4. Revisar los errores y corregir el CSV si es necesario.
  5. Hacer clic en "Confirmar Importacion" para cargar las filas validas.
  6. El sistema muestra un resumen final: filas importadas, filas omitidas, errores detallados.

10.4 Procedimiento: Exportacion de Reportes PDF

El sistema genera PDFs profesionales del lado del cliente usando jsPDF:

  1. Acceder a Reportes y seleccionar la pestana deseada.
  2. Configurar el periodo usando los filtros de fecha.
  3. Verificar que los datos mostrados son los correctos.
  4. Hacer clic en el boton "Exportar PDF".
  5. El sistema genera el PDF con los siguientes componentes:
    • Header: Logo SITREP con barra verde institucional, titulo del reporte, periodo.
    • KPI Cards: Tarjetas con indicadores principales del periodo.
    • Tabla: Datos tabulares formateados con jspdf-autotable.
    • Footer: Numero de pagina, fecha de generacion, URL del sistema.
  6. El navegador descarga automaticamente el archivo PDF.

10.5 Procedimiento: Configuracion de Reglas de Alerta

Guia para configurar reglas de alerta efectivas:

  1. Acceder a Alertas desde el menu principal.
  2. Hacer clic en "Nueva Regla de Alerta".
  3. Completar los campos de la regla:
    • Nombre: Identificador descriptivo (ej: "Diferencia de peso mayor al 10%").
    • Evento disparador: Seleccionar el tipo de evento que activa la evaluacion.
    • Condicion: Definir la expresion que debe cumplirse.
    • Destinatarios: Seleccionar roles o usuarios especificos que recibiran la alerta.
  4. Activar la regla con el interruptor.
  5. Guardar. La regla comenzara a evaluarse inmediatamente.

15.5 Autenticacion y Seguridad — Detalle Tecnico

Flujo de Registro

  1. Validacion: Zod schema valida email, password (min 8 chars, 1 mayuscula, 1 numero), nombre, apellido, rol, cuit, empresa, telefono
  2. Password strength: Minimo 8 caracteres, al menos 1 mayuscula y 1 numero
  3. Hashing: bcryptjs.hash(password, 10) — 10 salt rounds
  4. Email verification: Genera token via crypto.randomBytes(32) → SHA256 hash → almacena hash en DB
  5. Estado inicial: Usuario creado con activo: false, requiere aprobacion del ADMIN
  6. Notificacion: Email con link de verificacion enviado via nodemailer SMTP

Flujo de Login

  1. Input: email|cuit + password
  2. Busqueda: prisma.usuario.findFirst({ where: { OR: [{ email }, { cuit }] } })
  3. Verificacion: bcrypt.compare(password, user.password)
  4. Checks: Verifica emailVerified === true y activo === true
  5. Tokens: Genera par JWT:
    • Access token: 24 horas, contiene { userId, rol, email }
    • Refresh token: 7 dias, almacenado con hash en tabla RefreshToken
  6. Respuesta: { accessToken, refreshToken, user: { id, email, nombre, rol, ... } }

Flujo de Refresh

  1. Frontend detecta 401 → envia refresh token a POST /api/auth/refresh
  2. Backend verifica token → check revoked === false y expiresAt > now
  3. Verifica usuario activo === true
  4. Genera nuevo par de tokens, revoca el refresh token anterior
  5. Cross-tab sync: listener storage event sincroniza tokens entre pestanas

Flujo de Reset de Password

  1. crypto.randomBytes(32) → genera token aleatorio
  2. Hash SHA256 del token almacenado en DB con expiracion 24 horas
  3. Email con link /reset-password?token={raw}
  4. En reset: verifica token via SHA256, valida nueva password, hash bcrypt, actualiza

Flujo de Deploy Atomico

  1. Build en cloud (GitHub Actions, Ubuntu):
    • npm ci — instalacion deterministica de dependencias
    • npm run build — compila TypeScript a dist/
    • Empaqueta: dist/ + package.json + package-lock.json + prisma/ + ecosystem.config.js
  2. Transfer: scp del tarball al VPS (/tmp/)
  3. Deploy script (/opt/scripts-cicd/deploy_sitrep_backend.sh):
    • Extrae a /var/www/sitrep-backend-releases/YYYYMMDD-HHMMSS/
    • Copia .env desde release actual
    • npm ci --production — instala solo dependencias de produccion
    • npx prisma generate — regenera binarios para RHEL (fix critico 2026-02-15)
    • Symlink: /var/www/sitrep-backend → nuevo release
    • pm2 restart sitrep-backend
  4. Health check: curl https://sitrep.ultimamilla.com.ar/api/health
    • Espera respuesta {"status":"ok","db":"connected"}
    • Si falla → ROLLBACK automatico al release anterior (2 segundos)
  5. Cleanup: Elimina releases antiguos (conserva ultimos 5)

Prisma Binary Fix (critico)

Problema: GitHub Actions corre en Ubuntu y genera binarios Prisma para debian-openssl-3.0.x. El servidor de produccion es RHEL y necesita rhel-openssl-3.0.x.

Solucion: El deploy script ejecuta npx prisma generate en el servidor, regenerando el Prisma Client con los binarios nativos correctos.

Binary targets en schema.prisma: ["native", "rhel-openssl-3.0.x"]
