CU-A01: Iniciar Sesión Administrador

Actor Principal Administrador SITREP
Descripción Permite al administrador SITREP acceder al sistema mediante autenticación segura con usuario, contraseña y token de seguridad (2FA).
Precondiciones • Usuario registrado en el sistema
• Usuario con rol de Administrador asignado
• Credenciales activas y no bloqueadas
Flujo Principal 1. El administrador accede a la URL del sistema
2. El sistema muestra formulario de login
3. El administrador ingresa usuario y contraseña
4. El sistema valida credenciales y solicita token 2FA
5. El administrador ingresa código de token
6. El sistema verifica token y crea sesión
7. El sistema redirige al dashboard de administración
Flujo Alternativo • 3a. Credenciales inválidas: sistema muestra error y permite reintentar (máx 3 intentos)
• 3b. Usuario bloqueado: sistema notifica y sugiere contactar soporte
• 5a. Token inválido: sistema permite reingresar código
Postcondiciones • Sesión iniciada correctamente
• Log de acceso registrado en auditoría
• Dashboard de administración visible
Prioridad Alta - Crítico

Actor Principal Administrador SITREP
Descripción Presenta al administrador un panel de control con indicadores clave (KPIs) de la gestión de residuos peligrosos en tiempo real.

Postcondiciones • Dashboard visualizado correctamente

CU-G03: Crear Manifiesto Electrónico

Actor Principal Generador de Residuos
Descripción Permite al generador crear un nuevo manifiesto electrónico de residuos peligrosos, declarando tipo, cantidad, origen, transportista y operador destino.
Precondiciones • Sesión de generador activa
• Al menos un transportista y operador habilitados disponibles
Flujo Principal 1. El generador selecciona 'Nuevo Manifiesto'
2. El sistema muestra formulario con datos del generador precargados
3. Selecciona tipo de residuo del catálogo
4. Ingresa cantidad y unidad de medida
5. Selecciona transportista habilitado de la lista
6. Selecciona operador destino de la lista
7. Ingresa observaciones si corresponde
8. El sistema valida datos ingresados
9. El generador puede guardar como borrador o continuar a firma
Flujo Alternativo • 3a. Tipo de residuo no encontrado: permite búsqueda
• 5a. Transportista no disponible para el residuo: sistema filtra compatibles
• 6a. Operador no recibe ese tipo de residuo: sistema advierte
• 8a. Datos incompletos: sistema indica campos faltantes
Postcondiciones • Manifiesto creado en estado borrador o pendiente de firma
• Número de manifiesto asignado
Prioridad Alta - Crítico

CU-G07: Firmar Manifiesto (Electrónica/Remota)

Actor Principal Generador de Residuos
Descripción Aplica firma al manifiesto. Soporta mecanismo interno (Usuario+Pass+Token) y orquestación para múltiples firmantes.
Precondiciones • Sesión activa
• Manifiesto completo y validado
• Token de seguridad disponible
Flujo Principal 1. Generador revisa el borrador y acepta la DDJJ
2. Sistema solicita método de autenticación de firma
3. Opción A (Estándar): Ingresa contraseña y Token (OTP/Físico)
4. Opción B (Remota): Invoca servicio de Firma Digital Remota (Nube)
5. Sistema estampa la firma, bloquea el documento y genera el QR
6. Notifica al Transportista
Flujo Alternativo Firma Conjunta: Si el sistema detecta que se requieren múltiples firmas (ej. Responsable Técnico + Legal), el estado pasa a "Pendiente de 2da Firma" antes de liberarse.
Postcondiciones • Manifiesto firmado digitalmente
• Estado actualizado
• Transportista notificado
• QR generado
Prioridad Alta - Crítico

CU-T03: Confirmar Retiro en Origen (Modo Desconectado)

Actor Principal Transportista
Descripción Registro de retiro en zona sin señal. La operación debe completarse localmente al 100%.
Precondiciones • Sesión activa (preferentemente app móvil)
• Manifiesto asignado en estado 'Pendiente de retiro'
• GPS activo
Flujo Principal 1. Transportista selecciona manifiesto en App
2. Verifica bultos y cantidades
3. Georreferenciación: App captura coordenadas GPS del dispositivo (independiente de la red)
4. Firma en Pantalla: Generador firma sobre el dispositivo del transportista
5. Persistencia Local: Si no hay red, la transacción se guarda en LocalDB encriptada
6. Sistema encola la sincronización para cuando se recupere conectividad
Flujo Alternativo • 3a. Cantidad diferente: registra observación obligatoria
• 4a. Sin GPS: permite ingresar dirección manualmente con advertencia
• 6a. Sin conexión: almacena localmente para sincronizar después
Postcondiciones • Retiro registrado con evidencia
• Estado actualizado
• Partes notificadas
Prioridad Alta - Crítico

CU-T09: Arquitectura Offline-First y Sincronización

Actor Principal Sistema (App)
Descripción Capacidad del motor de la App para gestionar la intermitencia de red sin detener la operación.
Precondiciones • App móvil instalada
• Datos previos sincronizados
Flujo Principal 1. App detecta pérdida de conectividad
2. Activa modo "Offline-First" transparente al usuario
3. Todas las validaciones (QR, Tipos de Residuo) se ejecutan contra la caché local
4. Al recuperar señal, el Service Worker dispara la sincronización en segundo plano
5. Resolución de conflictos automática (prioridad timestamp servidor)
Flujo Alternativo • 5a. Conflicto de datos: sistema presenta opciones al usuario
• 5b. Sincronización fallida: reintenta automáticamente
Postcondiciones • Operaciones registradas localmente
• Sincronización exitosa al reconectar
Prioridad Alta

CU-O03: Recepción con Validación QR/Offline

Actor Principal Operador
Descripción Ingreso a planta. Debe soportar lectura de QR incluso si la garita de acceso no tiene red estable.
Precondiciones • Sesión activa
• Manifiesto en estado 'Entregado - Pendiente recepción'
• Transporte presente en planta
Flujo Principal 1. Transporte arriba a planta
2. Operador escanea QR del manifiesto
3. Validación: App verifica integridad del QR y estado "En Tránsito". Si no hay red, valida contra lista de "Esperados" descargada previamente
4. Registra ingreso y deriva a báscula
Flujo Alternativo • 3a. QR ilegible: ingreso manual
• 4a. Datos no coinciden: registrar observación
Postcondiciones • Recepción iniciada
• Proceso de verificación en curso
Prioridad Alta - Crítico

CU-O09: Cerrar Manifiesto

Actor Principal Operador
Descripción Finaliza el ciclo completo del manifiesto con firma de disposición final, completando la trazabilidad.
Precondiciones • Tratamiento registrado
• Proceso completo
Flujo Principal 1. Selecciona manifiesto a cerrar
2. Sistema verifica que todos los pasos están completos
3. Operador confirma finalización
4. Sistema solicita firma electrónica
5. Aplica firma de cierre
6. Sistema cambia estado a 'Cerrado'
7. Sistema genera certificado de disposición
8. Sistema notifica a todas las partes
Flujo Alternativo • 2a. Pasos incompletos: sistema indica qué falta
Postcondiciones • Manifiesto cerrado definitivamente
• Certificado generado
• Ciclo de trazabilidad completado
Prioridad Alta - Crítico

CU-S04: Registrar Auditoría

Actor Principal Sistema
Descripción Log automático de todas las operaciones realizadas con timestamp, usuario, IP y datos afectados.
Precondiciones • Operación realizada en el sistema
Flujo Principal 1. Usuario realiza cualquier operación
2. Sistema captura: fecha/hora, usuario, IP, acción, módulo
3. Captura datos antes y después del cambio
4. Almacena en tabla de auditoría
5. Registro inmutable
Flujo Alternativo • Error de registro: almacena en cola para retry
Postcondiciones • Operación auditada
• Registro inmutable disponible
Prioridad Alta - Crítico

CU-S08: Detectar Anomalías en Transporte

Actor Principal Sistema
Descripción Identifica automáticamente desvíos de ruta, tiempos excesivos y otros comportamientos anómalos.
Precondiciones • Manifiesto en tránsito
• Tracking GPS activo
Flujo Principal 1. Sistema recibe actualización de posición GPS
2. Compara con ruta esperada
3. Calcula tiempo transcurrido vs estimado
4. Si detecta anomalía, genera alerta
5. Notifica a SITREP y actores involucrados
6. Registra incidencia en el manifiesto
Flujo Alternativo • Falsa alarma: administrador puede descartar con justificación
Postcondiciones • Anomalías detectadas
• Alertas generadas si corresponde
Prioridad Alta
